# VPS Deployment Runbook

Date: 2026-06-24

## Scope

This runbook covers the Terminal VPS deployment path for the KMT Legal MVP. PLAN-26 adds separate panel-aware setup planning for aaPanel and compatible cPanel accounts. This runbook assumes a Node.js production process behind Nginx, PostgreSQL, private VPS filesystem uploads, disabled SMTP placeholders, and the AI Provider Gateway.

## Files

- Environment template: `deploy/env.production.example`
- One-command installer: `deploy/install/install.sh`
- Nginx template: `deploy/nginx/kmt-legal.conf.example`
- systemd template: `deploy/systemd/kmt-legal.service.example`
- Main handoff: `PROJECT_GUIDE.md`
- Panel compatibility plan: `docs/PLAN_26_PANEL_INSTALLER.md`

## Production Environment

Create `/etc/kmt-legal/kmt-legal.env` from `deploy/env.production.example` and replace placeholders in the VPS secret store or root-owned env file.

Required production decisions:

- `NODE_ENV=production`
- `APP_ENV=production`
- `APP_ORIGIN=https://...`
- `SESSION_COOKIE_SECURE=true`
- `CSRF_STRICT_ORIGIN=true`
- `STORAGE_DRIVER=vps-filesystem`
- `UPLOADS_DIR=/var/lib/kmt-legal/uploads`
- `MAX_UPLOAD_MB=5`
- `AI_PROVIDER=mock` or a configured provider through the gateway

Never set production demo values for:

- `KMT_DEMO_PASSWORD`
- `KMT_DEMO_TOTP_SECRET`

PLAN-25 installer values:

- `STAFF_2FA_MODE=disabled`
- `INSTALLER_ENABLED=true` only during first setup
- `INSTALLER_SETUP_TOKEN` generated by the install script
- `INSTALLER_LOCK_PATH=/var/lib/kmt-legal/install.lock`

After `/install` is locked, run `sudo kmt-legal-disable-installer` so `INSTALLER_ENABLED=false`.

## One-Command Install

On a fresh Ubuntu VPS, from the uploaded/cloned project directory:

```bash
sudo bash deploy/install/install.sh
```

The script installs OS dependencies, configures PostgreSQL, writes the production env file, builds the app, runs migrations/production seed, configures systemd/Nginx, and prints:

```text
https://your-domain.example/install?token=...
```

Open that URL, run preflight, create the first Super Admin, lock the installer, then run:

```bash
sudo kmt-legal-disable-installer
```

Do not run this root installer inside aaPanel/cPanel unless you intentionally bypass the panel and manage the VPS manually. Panel-managed deployments must use the PLAN-26 assisted path so the panel keeps ownership of domain, SSL, reverse proxy, database, and process-manager settings.

## Upload Storage

Create the private upload directory on the VPS:

```bash
sudo mkdir -p /var/lib/kmt-legal/uploads
sudo chown -R kmt-legal:kmt-legal /var/lib/kmt-legal
sudo chmod 750 /var/lib/kmt-legal /var/lib/kmt-legal/uploads
```

Nginx must not serve this directory. Downloads must go through `/api/files/[documentId]/download`.

## Build And Start

On the VPS release directory:

```bash
npm ci
npm run db:generate
npm run build
npm run db:migrate
sudo systemctl restart kmt-legal
```

Use `npm run db:migrate` for production migrations. `npm run db:migrate:dev` is for local development only.

## Nginx

Use `deploy/nginx/kmt-legal.conf.example` as the starting point.

Required checks:

- TLS is configured.
- Proxy points to `127.0.0.1:3000`.
- Upload request body limit allows multipart overhead while app enforces 5MB per file.
- `/uploads/` returns 404.
- HSTS remains commented until HTTPS renewal and first deploy are verified.

## systemd

Use `deploy/systemd/kmt-legal.service.example` as the starting point.

Required checks:

- Service runs as a non-root `kmt-legal` user.
- `EnvironmentFile` points to `/etc/kmt-legal/kmt-legal.env`.
- `ReadWritePaths` includes the private upload directory.
- `NODE_ENV=production` is set.

## Pre-Deploy Checklist

Run locally or in CI:

```bash
npm run qa:local
```

Run on staging after PostgreSQL is ready:

```bash
npm run db:migrate
npm run qa:local:e2e
```

Run network-dependent audit where registry access is available:

```bash
npm run security:audit
```

## Smoke Tests

After deploy:

- `GET /` returns 200 and includes security headers.
- `GET /login` returns 200.
- Anonymous `GET /admin` redirects to `/login?next=/admin`.
- Cross-origin `POST /api/auth/logout` returns 403.
- Staff login reaches `/admin` without 2FA while `STAFF_2FA_MODE=disabled`.
- `/login/2fa` and 2FA API routes remain disabled.
- Booking creates a consultation request.
- Admin converts consultation to case.
- Client sees only own portal data.
- Valid upload under 5MB succeeds.
- Oversized or unsupported upload fails.
- Authorized download returns `Content-Disposition: attachment`.

## Backup

Back up both PostgreSQL and uploads.

Example PostgreSQL backup:

```bash
pg_dump "$DATABASE_URL" > "kmt-legal-$(date +%Y%m%d%H%M%S).sql"
```

Example upload backup:

```bash
tar -czf "kmt-legal-uploads-$(date +%Y%m%d%H%M%S).tgz" /var/lib/kmt-legal/uploads
```

Store backups outside the app server when possible. Restore to staging first before touching production.

## Rollback

1. Stop deploy.
2. Restart the previous app release if the database migration is backward compatible.
3. If migration is not reversible, use a forward fix.
4. Do not delete uploads during rollback.
5. Re-run smoke tests.
6. Record incident notes with release id, request ids, timeframe, and affected role/resource.

## Incident Notes

For security or privacy incidents:

- Freeze deploys.
- Preserve logs and audit records.
- Rotate affected secrets.
- Check audit logs for affected users/resources.
- Prepare legal/client notification only through the approved office process.
